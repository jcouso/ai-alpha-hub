## 🧑‍💻 Coding Tips

### 1) Audit your AI chatbot APIs for injection attacks (post-McKinsey hack)
The McKinsey breach exploited unauthenticated SQL injection via an API endpoint. If you're building any LLM-backed feature with database access, here's a quick security checklist:

```bash
# Scan your API endpoints for injection vulnerabilities
# Using sqlmap (install: pip install sqlmap)
sqlmap -u "https://your-app.com/api/chat?query=test" \
  --level=3 --risk=2 --batch \
  --output-dir=/tmp/sqlmap-audit

# Or if you have Codex Security (free 30 days):
curl https://api.openai.com/v1/codex/security/scan \
  -H "Authorization: Bearer $OPENAI_API_KEY" \
  -d '{
    "repository": "https://github.com/your-org/your-repo",
    "scan_type": "api_security",
    "focus": ["injection", "authentication", "authorization"]
  }'
```

And in your codebase, always parameterize queries — never interpolate user (or agent) input:

```typescript
// ❌ VULNERABLE (this is what got McKinsey)
const result = await db.query(`SELECT * FROM messages WHERE content LIKE '%${userInput}%'`);

// ✅ SAFE — parameterized query
const result = await db.query(
  'SELECT * FROM messages WHERE content LIKE $1',
  [`%${userInput}%`]
);
```

### 2) Protect your GitHub Actions from AI agent attacks (post-Hackerbot-Claw)
The Hackerbot-Claw attack exploited GitHub Actions workflows. Lock down your CI:

```yaml
# .github/workflows/secure-ci.yml
name: Secure CI
on:
  pull_request:
    types: [opened, synchronize]

permissions:
  contents: read  # NEVER give write to PR workflows
  # pull-requests: write  # Only if you NEED PR comments

jobs:
  build:
    runs-on: ubuntu-latest
    # CRITICAL: Don't run on forks without review
    if: github.event.pull_request.head.repo.full_name == github.repository
    steps:
      - uses: actions/checkout@v4
        with:
          persist-credentials: false  # Don't leak tokens
      - name: Build & Test
        run: npm ci && npm test
        env:
          # Never expose secrets to PR builds from forks
          NODE_ENV: test
```

Key rules: `permissions: contents: read` (never write on PR workflows), `persist-credentials: false`, and gate fork PRs behind manual approval.
