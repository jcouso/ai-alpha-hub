### 2. Ladybird Browser Adopts Rust ‚Äî Using Claude Code + Codex for the Port ü¶Äüî•üî•
**Score: 9.5/10** | CONFIRMED | HN #2 at 988 points, 543 comments

The Ladybird browser project ‚Äî the independent, open-source browser engine started by former SerenityOS creator Andreas Kling ‚Äî announced it is adopting Rust to replace C++. The headline: they used AI to do the translation.

**The numbers:**
- 25,000 lines of Rust translated from C++
- 2 weeks of work (estimated months without AI)
- 52,898 test262 tests: **0 regressions**
- 12,461 Ladybird regression tests: **0 regressions**
- Zero performance regressions on JS benchmarks

**The method:**
- Used Claude Code and OpenAI Codex
- "Human-directed, not autonomous code generation"
- Hundreds of small prompts, steering agents through the translation
- Multiple passes of adversarial review using different models
- First target: LibJS (lexer, parser, AST, bytecode generator)
- Requirement: byte-for-byte identical output from both pipelines

**Why they switched from Swift to Rust:**
- Swift's C++ interop "never quite got there"
- Platform support outside Apple ecosystem was limited
- Rust ecosystem more mature for systems programming
- Many contributors already know Rust
- Firefox and Chromium have already begun introducing Rust

The blog acknowledges the result has a "translated from C++ vibe" ‚Äî intentionally mimicking C++ patterns for correctness verification first, with idiomatic Rust improvements planned later.

**Why this matters for builders:** This is the proof point. If a browser engine's JS pipeline can be ported from C++ to Rust with zero regressions in 2 weeks using AI, then massive code modernization projects are now feasible. Combined with the IBM/COBOL story below, this validates AI-assisted code translation as a production-ready technique. The "hundreds of small prompts" approach aligns perfectly with Boris Tane's structured workflow.

### Learn from Ladybird: AI-Assisted Code Translation at Scale
The Ladybird team's approach to porting C++ to Rust is the gold standard:
1. **Pick self-contained targets** ‚Äî Start with modules that have extensive test coverage (their JS engine had 52K tests)
2. **Human-directed, hundreds of small prompts** ‚Äî Not one big "translate everything" prompt
3. **Byte-for-byte verification** ‚Äî Require identical output from both pipelines
4. **Adversarial review** ‚Äî Use different models to analyze the translated code for mistakes
5. **Correctness first, idioms later** ‚Äî The initial translation intentionally mimics the source language's patterns

### The CLAUDE.md Decision Framework
After Theo's counter-take today, here's how to decide:

**Use structured context (CLAUDE.md, AGENTS.md, research.md):**
- Large codebases (>100 files)
- Team projects where multiple developers use the agent
- Repos with non-obvious architectural decisions
- Long-running feature development

**Minimize context files:**
- Small personal projects
- Well-documented codebases with clear conventions
- Prototyping and exploration
- When the maintenance burden exceeds the benefit

If you have a C++/Java/COBOL modernization project:
1. Find modules with extensive test coverage
2. Use Claude Code for translation with small, directed prompts
3. Require identical behavior verification
4. Use a different model for adversarial review
