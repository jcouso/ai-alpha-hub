## 🧪 Research Frontier

### ETH Zurich: AGENTS.md Files May Hinder AI Coding Agents
A new ETH Zurich paper concludes that despite widespread industry recommendations, auto-generated AGENTS.md context files may actually hurt AI coding agent performance. The researchers recommend:
- **Omit LLM-generated context files entirely** — they often add noise
- **Limit human-written instructions** to short, specific rules
- The sweet spot is concise human-authored guidance, not comprehensive auto-generated documentation

**Alpha angle:** This directly contradicts the industry consensus. If you maintain CLAUDE.md, AGENTS.md, or similar files, audit them for bloat. Shorter may be better. Test with and without.

- Sources: [InfoQ](https://www.infoq.com/news/2026/03/agents-context-file-value-review/)

### HN Signal: "LLMs Work Best When Users Define Acceptance Criteria First"
A Katana Quant blog post arguing that defining acceptance criteria upfront dramatically improves LLM coding output hit #3 on HN (252 points, 190 comments). This aligns with the AGENTS.md research — the problem isn't context quantity, it's context quality and specificity.

**Alpha angle:** Frame your coding agent prompts as spec + acceptance criteria, not open-ended descriptions. Think test-driven prompting.

## 🧑‍💻 Coding Tips

### 1) Use Claude for security auditing your own codebase
Inspired by the Firefox partnership — you can run a similar (scaled-down) analysis today:

```bash
# Feed your codebase to Claude Code for a security review
claude -p "You are a security researcher. Analyze this codebase for:
1. Memory safety issues
2. Input validation gaps
3. Authentication/authorization flaws
4. Logic errors that fuzzing wouldn't catch
For each finding, provide a minimal repro and severity rating." \
  --allowedTools Read,Bash \
  src/
```

### 2) Try the Figma MCP server for design-to-code
```json
// Add to your .vscode/mcp.json or claude_desktop_config.json
{
  "mcpServers": {
    "figma": {
      "command": "npx",
      "args": ["-y", "@anthropic-ai/figma-mcp-server"],
      "env": {
        "FIGMA_ACCESS_TOKEN": "your-figma-token"
      }
    }
  }
}
```
Now in Copilot/Claude Code: "Pull the design from Figma frame X and implement it as a React component" — or reverse: "Send this rendered component back to Figma as an editable frame."

### 3) Test-driven prompting pattern (from today's HN discussion)
```markdown
## Task
Build a rate limiter middleware for Express.

## Acceptance Criteria
- [ ] Uses sliding window algorithm (not fixed window)
- [ ] Configurable: windowMs, maxRequests, keyGenerator
- [ ] Returns 429 with Retry-After header
- [ ] Works with Redis for distributed deployments
- [ ] Passes: `npm test` with >90% coverage
- [ ] Types: full TypeScript, no `any`

## Anti-patterns to avoid
- No in-memory-only storage (must support Redis)
- No blocking the event loop
```
Frame your agent prompts this way. The ETH Zurich research + HN discussion both confirm: explicit acceptance criteria >> verbose context files.
